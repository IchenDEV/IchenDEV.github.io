软件开发这条主线，在过去的十几年里，脉络一直清晰可见：我们追求更棒的编辑器，更顺手的协作平台，以及更自动化的流水线。作为开发者，日常工作流程早已刻画在脑海中：打开 VS Code，敲代码，跑测试，提交到 GitHub，然后静候 Code Review，接着就是 CI/CD 的自动化部署。整个过程稳定而清晰，也正是它塑造了我们现代工程师特有的工作习惯。

然而，AI 的出现，却让这一切开始朝着一个全新的方向演变。最初，它只是个更智能的“补全器”，帮我们补全函数、生成那些重复的样板代码，甚至代笔测试用例。接着，它悄然融入编辑器，开始理解我们项目的结构。再后来，它的身影出现在终端、云端沙箱、Issue、Pull Request，乃至部署预览中，逐渐承担起更完整的任务。到了今天，AI 已经能将 Linear、Slack、GitHub、测试环境、浏览器、部署平台和监控系统这些环节串联起来。它不再仅仅是一个“会回答问题的模型”，而是逐渐蜕变，成为一个能深度参与软件交付的工程系统。

## **上古时代**：编码入口被打开，但上下文的空洞依然存在

在我看来，VS Code、GitHub 和 Copilot，这三者构成了我们技术叙事中所谓的“**上古时代**”的基础设施。

VS Code 的出现，彻底改变了我们本地开发的体验。它巧妙地将编辑、调试、插件、终端和语言服务整合到一个轻量级的环境里，让程序员的日常工作流程变得前所未有的统一和顺畅。而 GitHub 呢，它重塑了协作的方式。代码不再仅仅是躺在你本地硬盘里的文件，它摇身一变，成了可以被讨论、审查、追踪、自动化测试和持续交付的社会化资产。至于 Copilot，它更是意义非凡，第一次将 AI 以如此大规模的方式，融入到我们最核心的编码动作之中。

不得不说，这一代工具确实解决了许多实实在在的难题。面对空白文件，我们不再那么手足无措；那些重复的代码，也不再那么消耗我们的宝贵时间。无论是 API 调用、类型定义、测试样板，还是各种配置文件，都能以更快的速度生成。这意味着，开发者们终于可以将更多的心力投入到设计和调试这些更具创造性的工作上，而不是被机械的输入所困扰。

然而，它们的边界也清晰可见。Copilot 式的补全，主要还是围绕着当前文件、当前函数、以及光标附近的语境展开。它很擅长“顺着你正在写的东西往下写”，这一点毋庸置疑。但要让它真正理解一个产品设计的深层原因，或者跨越几十个文件去完成一个完整的功能，那可就太难了。它可以帮你敲出一段代码，却很难回答“这个功能到底需要改动哪几层？”“这样做会不会破坏现有的权限模型？”又或者“这个变更对计费、日志、测试、灰度会产生什么影响？”这样的复杂问题。

所以，在我看来，**上古时代**的核心成就，在于极大地提升了个人编码的速度。但它的核心局限，也恰恰在于上下文理解得太浅。它确实将开发者从一部分重复劳动中解放了出来，但那些更深层次的工程判断、任务拆分、跨文件修改，以及至关重要的质量保证，依然沉甸甸地压在我们人类的肩上。

这个阶段的 AI，更像是一盏紧贴着光标的小巧聚光灯。它能精准地照亮局部，让手边的代码更容易被写出来。然而，它却无法照亮整个系统。它知道你正在写什么代码，却未必能理解这个模块存在的真正意义。它能帮你补齐一段函数，却难以洞悉一个产品从需求萌芽到最终交付的完整路径。
## **封建时代**：当编辑器化身为对话式工作台

Cursor、Trae、Windsurf，这些工具的出现，标志着 AI 在编程领域的角色发生了质的飞跃。它不再仅仅是光标旁那个默默补全代码的小助手，而是摇身一变，成了编辑器里的全能协作者。它们身上有个共通的印记：AI 不再满足于帮你补上一行代码，而是能围绕整个项目和你对话，帮你检索文件，修改多个文件，甚至能解释那些让人头大的错误，并提出一整套实现方案。

我个人觉得，这一阶段的演变意义深远，因为它第一次把“对话”推到了开发的入口。想想看，以前我们写代码，总得先知道该打开哪个文件，再琢磨着怎么修改；现在呢，你可以直接描述你的目标，让 AI 来帮你找出相关的模块，解释复杂的调用链，甚至为你构思出改动方案。开发者的起手式，就这样从“定位代码”变成了“描述意图”。

这种转变，带来了前所未有的新鲜感，但也隐约带着一丝危险的轻盈。你随口一句“帮我实现登录后的会员权益页”，AI 可能真的就能帮你创建页面，补上接口调用，生成状态处理逻辑，连错误文案都替你考虑周全。它就像一个对框架了如指掌的初级工程师，手脚麻利，也乐于尝试。

然而，即便身处这个**封建时代**，这些工具也并非没有烦恼，它们仍然面临着几道难题。

首先是**长程上下文**的挑战。项目越大，AI 越容易在那些细微之处、那些不成文的约定上栽跟头。它可能把页面改得天衣无缝，却漏掉了重要的埋点；接口调用写得漂亮，却忘了接入权限校验；修了一个 bug，却不小心又引入了另一个边界问题。这就像一个人在迷宫里，只顾着眼前的路，却忘了整个迷宫的布局。

接着是**工程集成**的困境。在一个真实的开发项目里，我们有 lint 检查、类型系统、测试用例、构建流程、环境变量、数据库迁移、灰度发布、日志规范……这些都是工程的血肉。AI 固然能写代码，但它并不天然理解这些工程上的约束。它或许能生成一个看起来正确的实现，但它未必懂得团队为何要这样组织目录结构，为何某个接口必须保证幂等，又为何某个错误不能直接暴露给用户。这些都是代码背后的人情世故。

还有**多人协作**的复杂性。一个 AI agent 在你的本地分支上改代码，这事儿还算简单。但当多个开发者、多个 agent、多个需求同时推进时，冲突、代码审查、合并、回滚，这些都会变得异常复杂。传统的 Git 分支尚能承载协作的重担，但 AI 那高并发的产出能力，无疑让分支管理的压力陡然增加。

尽管如此，**封建时代**的价值依然不容忽视。它让“AI 编程”第一次真正具备了产品形态，从概念走向了现实。但它也清晰地揭示了一个事实：仅仅把 AI 塞进编辑器，是远远不够的。AI 若想真正融入工程实践，我们还需要任务环境、执行环境、协作环境和审查环境同步进行变革。这就像为一艘强大的新船，打造一个全新的港口生态。
## 编程助手的进化：从建议到执行

Claude Code 和 Codex 的出现，标志着编程助手迈入了新的阶段。它们不再仅仅停留在编辑器内的辅助，而是更进一步，开始**面向任务执行**。

这可不是小打小闹，这意味着 AI 编程助手正悄然向着“接手任务”的方向演进。你瞧，它不再只是帮你补全个把函数，而是能真正承担起一个完整的 Issue：比如修复导出失败的 bug，补齐支付回调的幂等逻辑，甚至重构某个组件，增加单元测试，或者评估一次迁移的潜在风险。它会像个真正的工程师一样，先读取项目上下文，然后提出解决方案，着手修改代码，运行必要的命令，最后把成果递到你面前，等待你的审查。

在我看来，这一阶段最核心的突破，在于 AI 终于获得了**行动能力**。这种行动力，具体体现在三个方面：它能理解项目的来龙去脉，能灵活调用各种工具完成任务，更厉害的是，它还能对自己的产出进行初步的自检。当然，模型本身的强大依然是基石，但真正让体验脱胎换骨的，是模型、工具、环境和流程之间珠联璧合的协同。

然而，能力越大，责任和风险也随之升级。一旦任务型 Agent 拥有了运行命令、修改文件、乃至操作 Git 的权限，其潜在的风险就远超简单的代码补全。它可能会误删文件，对环境产生错误的理解，甚至生成看似合理却难以维护的架构。更麻烦的是，它还会遇到各种平台兼容性问题：本地开发环境和云端环境的差异，私有依赖安装失败的窘境，数据库连接受限的无奈，测试数据缺失的尴尬，以及权限不足的掣肘。我们不得不承认，AI 越是深入真实的工程实践，就越需要严谨的工程化约束来保驾护航。

所以，我们不妨把**近代阶段**的主题概括为：**AI 开始真正动手做事，而人类的角色，则从纯粹的代码编写者，转变为任务的设计师、成果的审查者和风险的控制者。** 这无疑拔高了我们人类开发者工作的重心。你的目光不再仅仅聚焦于每一行代码如何书写，而是要更宏观地思考：任务该如何拆解？权限该如何分配？产出该如何验证？万一失败了，又该如何优雅地回滚？这真是一个充满挑战也充满机遇的时代。
## 现代AI开发的核心：把一个人变成一支小型工程队

进入**现代阶段**，Codex App、Superconductor、T3 Code、Cmux、Cursor 3 这些工具的出现，让我们的思考维度变得更广。我们不再仅仅纠结于“一个Agent能否独立完成任务”，而是更深入地探讨“多个Agent如何协同工作”、“怎样有效隔离上下文”、“如何对比不同方案的优劣”，以及“如何将AI的产出无缝融入团队的工程实践”。

想象一下，一个更贴近真实的工作场景是这样的：开发者面前是一个统一的工作台，左侧清晰展示着Git分支和变更记录，中央是多个Agent的活跃会话，右侧则集成了文件树、测试面板、运行反馈和提交入口。在这里，每一个Agent都像一位临时加入的工程伙伴，各司其职，负责相对明确的任务。而作为人类开发者，我们则站在这个枢纽，负责擘画整体目标、巧妙分配任务、严格检查成果、谨慎合并代码，并始终掌控风险。

这种工作流的核心转变在于，我们不再把AI仅仅看作一个“自动补全”的工具。更恰当的视角，是将其视为一组可以灵活调度的执行单元。一个Agent或许专注于解读需求文档，另一个则负责修改核心代码，再一个可能忙于补充测试用例，还有的Agent则扮演审查者的角色，或者深入文档、查找历史约定。人类开发者无需亲手敲下每一行代码，但必须牢牢把握方向。我们可以让AI跑得飞快，但绝不能让它脱缰狂奔。

通常，现代开发的第一步，并非急于动手写代码，而是先厘清工程的边界。比如，在项目根目录放置`AGENTS.md`、`CLAUDE.md`、`README.md`、`OpenSpec`这类规范文件，它们就像是给AI的“项目说明书”。里面需要详细阐述项目结构、技术栈、包管理方式、测试命令、提交规范、安全边界、命名规则、目录职责，以及哪些文件是绝不能随意改动的“禁区”。

过去，这些宝贵的信息往往通过老员工的口头传授来维系。而现在，我们需要将其沉淀为AI能够理解的文本。若缺少这些明确的指引，Agent便只能依靠猜测来工作。在演示阶段，猜测或许显得效率惊人，但在真实的工程实践中，它却极易埋下难以预料的隐患。

接下来，我们需要将一个需求拆解成一个个可执行的小任务。在现代AI开发中，最怕的就是把一句笼统的话直接抛给Agent，比如“帮我把整个会员系统做完”。这样的任务范围太大，AI很容易迷失方向，开发者也难以有效审查。更明智的做法，是将任务细分为几个清晰的单元：首先，读取现有的账号模块和权限模块；接着，设计会员状态字段；然后，实现后端接口；再编写前端页面；别忘了补充测试；最后，检查埋点和错误处理，并生成变更说明。

每个任务都应该有明确的输入、输出和验收标准。AI非常擅长执行明确的指令，但它并不能替你承担因目标模糊而带来的责任。

再进一步，为每个任务开辟独立的“工作空间”至关重要。这时，`git worktree`的价值便凸显出来。在传统的开发模式中，许多人习惯于在同一个仓库里切换分支，频繁地stash和恢复环境。然而，当AI加入后，这种方式很快就会变得一团糟。因为你可能同时让三四个Agent在不同方向上进行尝试，如果它们都在同一个目录下修改文件，冲突的解决将变得异常棘手。

`worktree`的妙处在于，同一个仓库可以衍生出多个独立的目录，每个目录对应一个分支，每个Agent都在自己的“专属领地”里工作。如此一来，一个Agent专注于修复支付问题，另一个修改前端交互，还有一个补充测试，它们之间互不干扰。最终，你只需比较每个分支的差异，决定哪些内容值得合并到主线。

我们发现，让**Agent先读，再写**，是一个更稳妥的策略。许多AI工作流之所以失败，并非模型能力不足，而是因为它一上来就急于修改代码。在现代开发中，一个更为稳健的流程是先让Agent进行“代码侦察”。你可以要求它首先回答：这个功能会涉及哪些目录？核心入口在哪里？已有的模式是什么？哪些测试可能会受到影响？哪些文件是绝不能触碰的？然后，让它提交一个简短的实现计划。只有当这个计划看起来合理时，才允许它进入真正的修改阶段。

这个步骤能显著降低随意修改的风险。它也让开发者在代码真正被触动之前，就能判断AI是否真正理解了项目。

接下来，可以将实现任务交给一个主Agent，同时安排旁路Agent进行检查。例如，你要修改一个“项目技术选型”的OpenSpec变更，主Agent负责完成文档和任务文件，而旁路Agent则专注于审查：有没有违反OpenSpec的结构？是否遗漏了`proposal.md`、`tasks.md`、`design.md`？有没有破坏已有的约定？是否引入了不必要的架构变更？

这类旁路审查非常有用，因为AI产出速度越快，人类的审查压力就越大。让另一个Agent先行进行一轮机械检查，能把许多低级问题挡在前面。这样，当人类开发者查看最终差异时，注意力就能更集中于设计质量和产品判断上。

第六步，要求Agent在每次修改后提供验证结果。现代AI开发不能只听它一句“改好了”。你需要让它明确运行了哪些命令，以及这些命令的输出是什么。比如，`pnpm lint`是否通过，`pnpm test`是否通过，类型检查是否通过，OpenSpec校验是否通过，构建是否通过。如果有些命令没有运行，原因是什么？如果测试失败，是原有失败还是本次改动导致？

这里有个很重要的习惯：我们不要仅仅满足于结论，更要看具体的命令和结果。Agent的自然语言总结听起来可能天衣无缝，但工程结果必须由实际运行的命令来证明。

第七步，用差异（diff）进行人工审查。AI完成修改后，开发者切勿急于提交。首先，检查文件数量，看看是否修改了不该动的文件；接着，深入核心文件，判断实现是否与现有结构保持一致；再查看测试文件，确认测试是否真正覆盖了本次变更；最后，审阅生成的文档，避免为了完成任务而写出空泛的描述。

现代工具通常会将“Files”、“Changes”、“Checks”整合在一起，正是为了让开发者在一个界面内完成这些工作。AI可以生成代码，但最终进入代码仓库的，必须是经过人类严格审查的代码。

第八步，将AI的工作产物写回工程系统。例如，本次任务对应一个OpenSpec变更，那么就要让它更新`tasks.md`的完成状态；如果修改了接口，需要更新接口说明；如果新增了行为，要补充测试说明；如果存在迁移风险，则需写入变更备注。过去许多工程团队的文档之所以滞后，往往是因为编写文档耗时耗力。现在，AI能够承担大部分文档整理工作，而开发者要做的，就是定义标准并检查其真实性。文档不应仅仅是“看起来很完整”的摆设，它应该精准地反映系统的真实状态。

第九步，在提交前进行一次清理和归档。在AI的工作流中，很容易产生大量的中间信息：计划、解释、失败尝试、临时文件、重复注释。提交前，我们可以让Agent帮助清理无关的改动，但这一步需要格外谨慎。可以先让它列出准备清理的内容，并说明清理的安全性，然后再执行。提交信息也可以让Agent生成，但人类必须将其修改为清晰的工程语言。一个好的commit message应该清楚地说明修改了什么、为什么修改、以及影响范围是什么。它不是写给机器看的，它是写给未来的自己和团队看的。

最后一步，是将多个Agent的结果合并成一个稳定的版本。并行开发的优势在于探索速度快，但代价是选择变得更多。你可能会得到三个看起来都能运行的实现方案。现代开发者的价值，恰恰体现在这里：你需要判断哪个方案更贴合项目结构，哪个方案长期维护成本更低，哪个方案测试更完善，哪个方案引入的新概念最少。AI可以为你提供更多的可能性，但它不会自动替你做出正确的工程取舍。最终合并到主分支的，不一定是代码最多的方案，也不一定是看起来最炫酷的方案，而应该是最能融入现有系统的方案。

## 现代AI工程队正在和真实工具链深度连接

现代 AI 开发，正悄然发生一个根本性变化：Agent 们不再只安居于代码仓库的一隅，它们开始与真实的、公司内部的工具链深度融合，打成一片。

回想一下，过去的开发流程是怎样的呢？任务的源头，往往是人。产品经理在 Linear 里敲下需求，运营或用户在 Slack 里反馈着问题，设计师在 Figma 上更新着页面，而工程师们则在 GitHub Issue 里记录着每一个 bug。我们开发者呢，就像个陀螺，不停地在这些工具间切换：看 Linear，翻 Slack 聊天记录，阅读文档，打开本地代码，启动开发环境，提交 PR，焦急等待测试结果，部署预览版，最后再回到 Slack 或 Linear 里同步最新进展。

当 AI 涉足工程领域，这条原本由人驱动的链路开始被重新编排。一个真正成熟的 Agent 工作流，绝不仅仅是会打开仓库写代码那么简单。它应该能从 Linear 领取任务，像个老兵一样，吃透 Issue 的背景信息，甚至能在 Slack 里检索到那些零散却关键的讨论。它会找到对应的设计稿或产品文档，然后才回到代码仓库里，精准定位到需要修改的地方。当修改完成，它还能像模像样地创建独立分支，启动测试环境，部署预览版，甚至能像人一样打开浏览器检查页面，运行自动化测试，最后把结果工工整整地写回 Linear 或 GitHub PR。

这意味着，AI 开发的边界，正从我们熟悉的“代码文件”向更广阔的“工作流系统”延伸。

不妨想象一个真实的任务场景：Linear 里躺着一个 ticket，上面写着“新用户注册后，会员权益页没有展示默认套餐”。Agent 拿到这个 ticket，会先像个侦探一样，提取出关键信息：问题发生在注册后，涉及会员权益页、用户初始化、套餐默认值、前端展示和后端接口。接着，它会去 Slack 里翻翻看，最近有没有类似的反馈，也许会发现客服同事们正抱怨着好几个新用户都遇到了空页面。然后，它打开代码库，开始在注册流程、会员状态初始化逻辑和权益页渲染逻辑之间穿梭。此刻，它不再是凭借一句 prompt 在盲目猜测，而是在真实的工具链中，有条不紊地收集着上下文。

下一步，它会在自己的 `git worktree` 里，悄悄创建一个独立分支。这样做的好处是显而易见的：主工作区保持干净整洁，其他 Agent 也能不受干扰地并行处理别的任务。它会修改后端初始化逻辑，把默认套餐状态补齐；再调整前端空状态的处理，最后还不忘补上测试用例。当第一轮实现完成，它会运行 `lint`、类型检查、单元测试以及相关的集成测试。如果项目配置了预览部署，它还会触发一次临时的环境部署。

此刻，`browser use` 和 `computer use` 的价值就彻底显现了。传统的 AI Coding 很多时候停留在“代码看起来写好了”这个阶段。但真实的产品开发，怎能只看代码？运行结果才是王道。Agent 可以像个挑剔的用户，打开预览环境，一步步走完注册流程：输入邮箱，完成注册，进入权益页。它会仔细检查默认套餐是否展示，按钮是否可点击，空状态是否消失，接口返回的数据是否正确。它甚至能截图，读取页面文本，将预期结果与实际结果进行比对。对于前端开发来说，这一步简直是救命稻海，因为很多问题，单凭代码 diff 是发现不了的，只有在浏览器里才能一览无余。

而 `computer use` 则更进一步，它让 Agent 拥有了操作更完整图形界面的能力。它可以打开本地应用，点击按钮，填写表单，切换页面，查看控制台报错，甚至能像个经验丰富的开发者一样打开开发者工具。在某些特定场景下，它甚至能操作第三方后台。这让 AI 从“代码作者”的角色，更接近于“测试同事”和“产品验收同事”。当然，这里我们必须强调权限边界，绝不能让 Agent 随意操作生产后台、真实支付、真实用户数据和那些不可回滚的配置，这是底线。

在**深度工具集成**之后，现代 AI 开发开始呈现出一种小型工程队的精悍面貌。

Linear 就像是任务的入口，所有工作都从这里开始。Slack 则像团队的集体记忆，也像问题爆发的现场。GitHub 毋庸置疑是代码协作的中心。CI 扮演着严谨的质量检查员，而部署平台则像一个个临时的实验室。浏览器和 `computer use` 则是最终的验收环境。Agent 在这些系统之间穿梭自如，将原本需要人反复搬运的信息，像一条条线索一样串联起来。而我们人类开发者，则站得更高，负责判断任务的价值，方案的合理性，权限的安全性，以及最终结果是否可以合并。

这与早期“让 AI 写一个函数”的工作方式，已经完全是两种境界。早期的 AI 编码，更像是你身边坐着一个聪明的助手，你问它某个函数怎么写，它便给出答案。而现代 AI Engineering，则更像是你在指挥一支精锐的临时工程队：有人从 Linear 领取任务，有人翻阅 Slack 了解背景，有人埋头写代码，有人跑着测试，有人部署预览版，还有人在浏览器里进行验收。每一个环节，虽然仍需要人类的监督，但信息流和执行流已经被大幅压缩，效率提升显著。

这里面最重要的，并非“AI 会不会调用更多工具”，而是“工具之间能不能形成闭环”。

如果 Agent 只会读 Linear，却不懂得把结果写回去，那它充其量只是多了一个输入源而已。如果它只会部署 preview，却不打开浏览器验证，那也不过是多了一次自动发布。如果它只会跑测试，却无法解释失败的原因，那我们人类依然需要做大量的后期处理。真正有价值的深度连接，是让任务从萌芽到完成，形成一条清晰可追踪的路径：任务来源一目了然，改动分支清清楚楚，测试结果明明白白，预览地址触手可及，验收记录有凭有据，PR 说明言简意赅，即使失败，原因也清晰可查。

这正是现代 AI 工程平台与普通 AI 编辑器之间的本质区别。普通编辑器主要致力于提升代码生成的效率，而现代 AI 工程平台，则致力于提升整个工程链路的吞吐量。它关心的不再是某一段代码写得有多快，而是从一个需求被提出，到它被验证、被合并、最终被发布，这中间有多少摩擦可以被消除，有多少低价值的沟通可以被自动化，又有多少关键的上下文信息能够被系统妥善保存下来。

然而，这种深度连接，也必然伴随着新的风险。

首先是**权限过大**的风险。一旦 Agent 能够读写 Slack、Linear、GitHub、部署平台乃至浏览器环境，它的权限就几乎等同于一个真实员工的工作权限。这意味着它可能接触到敏感的讨论、内部需求、用户反馈、密钥配置、测试账号以及业务数据。因此，权限必须严格按照任务授予，最好能遵循最小权限原则。如果只需要读取 ticket，就绝不赋予管理权限；能部署测试环境，就不要给生产发布权限；能访问假数据，就不要触碰真实用户数据。

其次是**上下文污染**的风险。Slack 里充斥着各种噪音，Linear 里的需求也可能不够完整，甚至有些历史讨论已经过时。如果 Agent 不加分辨地将所有信息都视为事实，就很容易做出错误的判断。一个成熟的系统，需要让 Agent 能够标注信息的来源：这个结论是来自哪条 ticket，哪个 Slack 讨论，哪个文档，还是哪个代码文件？这样，在人类审查时，才能判断它是否误读了上下文。

第三个风险是**自动验收的幻觉**。`Browser use` 可以点击页面，`computer use` 可以操作界面，但这并不等同于它真的理解了产品体验。它或许能确认页面没有报错，却不一定能判断交互是否自然流畅；它能确认按钮存在，却不一定能判断文案是否准确得体；它能走完既定路径，却不一定能判断商业逻辑是否成立。因此，自动验收更适合检查那些明确的标准，比如页面元素、接口返回、跳转路径、错误提示、视觉回归、表单提交等。而对于产品判断、体验判断和业务取舍，人类依然是不可或缺的。

第四个风险是，**流程被自动化后反而变得更难观察**。以前，人一步步地做事，虽然慢，但每一个动作都在我们脑子里留下了清晰的印记。Agent 一口气读任务、改代码、部署、验收、写 PR，如果中间的日志不够清晰，我们人类反而会更难知道它究竟做了什么。所以，现代 AI 工具必须高度重视过程记录：它读了哪些文件，调用了哪些工具，运行了哪些命令，部署到了哪个环境，浏览器验收走了哪些路径，失败后又修改了什么？

这也是为什么未来的 AI Engineering Platform 必须具备强大的“**可观察性**”。不只是线上系统要可观察，Agent 的行为同样要可观察。你需要能够清晰地看到它的任务计划

## AI Coding Agent正在从补全器成长为工程流程参与者

AI Coding Agent 的演进，在我看来，可以拆解成几个层次，就像一层层剥开洋葱，逐渐触及更深的核心。

最基础的一层，我们称之为 **Code Completion**。它要解决的，是让我们的手速跟上思维的速度。想象一下，你正在敲代码，Copilot 的行内建议悄然出现，就像一位默契的搭档，帮你填补那些重复的、模式化的代码。它让那些繁琐的重复劳动变得更少，让局部的实现过程变得更快，我们因此能更专注于核心逻辑。

再往上走，是 **Code Editing** 的领域。这时候，AI 不再满足于仅仅帮你补全一两句代码，它开始能够理解并修改一个函数、一个文件，甚至是一组相互关联的文件。像 Cursor、Windsurf、Trae 这样的工具，正是这个阶段的先行者。我们开始用更自然的语言告诉 AI 我们的意图，然后审视它给出的修改（那个熟悉的 diff 界面），再决定是否采纳。这是一种更深层次的协作。

接下来，我们看到了 **Task Agent** 的崛起。Claude Code、Codex、GitHub Copilot cloud agent 这些名字，开始围绕着完成一整个任务而努力。它们能像一个初级工程师一样，研究整个代码库，琢磨出一套实现计划，然后在一个新的分支上动手修改代码。当然，最终的决策权依然在我们手中，我们需要审查它们的成果，迭代，直到满意，最后才让它们创建 Pull Request。

而最终的愿景，是构建一个完整的 **Engineering Platform**。在这里，Agent 不仅仅是写代码的工具，它会深度融入整个工程流程，从需求分析，到项目管理，再到测试、部署、文档撰写、安全检查，甚至监控。在这个阶段，有几个核心概念我们不得不提：MCP、Skills、Harness、Subagents、Pipeline，它们共同构筑起这个宏大的图景。

**MCP (Multi-Contextual Processor)** 的价值，在于它能让 Agent 突破代码库的边界，连接到更广阔的外部世界。毕竟，对一个工程 Agent 来说，代码本身只是冰山一角。真正的任务，需要它能触及需求文档、设计稿、实时的监控数据、数据库结构、Issue 状态、用户反馈，甚至日志和部署系统。MCP 的出现，就像给 AI 装上了眼睛和耳朵，让它能在受控的环境下读取这些丰富的信息。同时，它也让那些零散的工具调用，变得像标准化接口一样规范和可靠。

**Skills** 的意义，在于将团队的经验沉淀为可复用的模块。设想一下，一个 Skill 可以打包一系列指令、资源，甚至是一些可选的脚本。这样一来，Agent 就能更可靠地执行特定的工作流程。我们可以把团队里那些宝贵的经验，比如“如何编写高质量的 API 测试”、“如何自动生成发布说明”、“如何进行安全审查”，甚至“如何严格按照公司组件规范来开发前端”，都沉淀给 Agent。这样，就不用每次都从头解释，Agent 也能像一个有经验的同事一样，按部就班地完成任务。

**Harness** 的存在，是为了提供一个受控的执行环境。一个好的 Harness，可不仅仅是简单地把模型连接到 shell 那么简单。它需要精心管理上下文，确保权限得当，监控命令的执行，处理错误恢复，记录详尽的日志，甚至还要考虑成本控制、设置审查点和终止条件。如果没有 Harness，Agent 很容易变成一个拥有过大权限的“脱缰野马”；有了它，Agent 才能被稳妥地纳入到整个工程体系之中，可控且安全。

**Subagents** 的概念，则体现了分工协作的智慧。我们可以设想一个主 Agent，它负责理解整体需求并协调各项工作。而在这个主 Agent 之下，可以有多个子 Agent 各司其职：有的专注于实现核心功能，有的负责编写和运行测试，有的专门进行代码审查，有的则专注于撰写文档，甚至还有的负责进行迁移检查。这就像一个微型的工程团队：有人负责具体的开发，有人负责质量把控，有人负责确保代码规范，有人负责把产品讲清楚，还有人负责排查潜在风险。它们共享同一个目标，但各自拥有相对独立的上下文和明确的任务边界。

当所有这些能力最终组合起来时，AI Coding Agent 就不再是我们最初认识的那个“会写代码的聊天机器人”了。它开始真正成为工程流程中不可或缺的参与者：它能读懂需求，拆解任务，研究现有代码，编写实现，跑测试，自动生成 PR，甚至能解释潜在的风险，补充缺失的文档，为发布做好准备。而我们人类工程师的工作，也随之发生了深刻的转变：我们不再需要逐行输入代码，而是将重心放在定义清晰的目标、设定合理的约束、仔细检查 Agent 的成果，以及维护整个系统的健康和一致性上。这无疑是一场工程范式的变革。


## 工具的演进：一场永无止境的迭代

工具，就像生命体，总是在不断地新陈代谢。每一次的更新，都像是对前一代工具所暴露问题的回应，然后，又不可避免地催生出新的挑战。

我们不妨回顾一下这条有趣的演进路径：

Copilot 曾以其局部编码的高效率，让开发者眼前一亮。但很快，它在项目整体上下文理解上的短板便暴露无遗。于是，Cursor、Windsurf、Trae 等工具应运而生，它们试图将 AI 的触角延伸到整个项目语境中，让 AI 不再是孤立的补全工具。

当 AI IDE 解决了编辑器内的上下文问题，我们又发现，仅仅理解上下文还不够，任务的实际执行能力显得捉襟见肘。这时，Claude Code、Codex 这样的工具登场了，它们让 AI agent 不仅能读文件、跑命令，甚至能修改代码，提交 PR，将 AI 的能力推向了更深层次的自动化。

然而，单任务 agent 的出现，虽然解决了特定任务的执行问题，却又凸显出多任务管理的复杂性。于是，Codex App、Superconductor、Cmux、Cursor 3 等工具开始崭露头角。它们不约而同地强调并行 agent、worktree、云端环境、统一工作区以及 agent 管理，试图为开发者提供一个更全面、更高效的协作平台。

并行 agent 的出现，确实提升了探索的效率，但随之而来的，却是质量控制、权限管理、成本控制和技能复用等一系列新的难题。这使得 MCP、Skills、Subagents、Harness 这些概念和工具变得日益重要，它们旨在为 AI 驱动的开发流程提供更精细化的管理和支持。

而**深度工具集成**，虽然在很大程度上弥补了上下文割裂的问题，却也抛出了权限、审计、环境隔离和自动验收等棘手的挑战。这让我不禁思考：未来的工程平台，恐怕不能仅仅停留在“连接”工具的层面。它必须同时具备强大的连接能力和严密的控制能力。只知道连接工具是远远不够的，我们还需要它能限制权限、记录过程、追踪来源、复现结果，甚至支持回滚。这听起来有些复杂，但却是我们不得不面对的现实。

这条工具演进的背后，其实有几个稳定且强大的驱动力在默默推动着。

首先，是**用户痛点**。开发者真正渴望的，并非仅仅是一个更会“聊天”的模型。他们更希望减少重复性的工作，避免频繁的上下文切换，摆脱环境配置的折腾，少一些低价值的排错，最终，能产出更多可靠的成果。这是一个非常朴素且真实的需求。

其次，是**技术瓶颈**。模型的上下文窗口大小、推理能力、工具调用方式、长任务的稳定性、延迟以及成本，都像无形的边界，决定着工具形态的演变。模型对长上下文的理解能力越强，工具就越能从简单的代码补全，走向复杂的任务执行；模型调用工具越稳定，agent 才能真正融入到我们的日常开发流程中。

再者，是**工程复杂度**。现代应用早已不是简单的前端加后端。它是一个庞大的生态系统，涵盖了账号、计费、权限、内容、推荐、AI 调用、日志、风控、监控、合规、灰度、增长实验等等。如果 AI 要深度参与构建，它就必须理解这套复杂的系统。仅仅依靠单点的代码补全，已经无法满足需求，工程平台化，才是大势所趋。

我们也不能忽视**团队协作**的重要性。一个人用 AI 写个 demo 固然轻松，但一个团队要持续维护一个产品，用 AI 辅助开发，这其中的难度可想而知。真正的挑战在于如何保持代码风格的一致性、架构边界的稳定性、测试覆盖的可靠性、PR 审核的可行性、部署的可控性以及事故的可追溯性。AI 工具最终要服务于整个团队，而不仅仅是满足个人一时的兴奋感。

最后，是**组织效率**。我们发现，许多工程上的消耗，并非发生在写代码本身。它更多地体现在信息转移、状态同步、问题复现、测试验证、环境部署以及上线沟通这些环节。如果现代 AI 工具仅仅能提升代码生成的速度，那么它的收益很快就会触及天花板。只有当它能够连接任务、讨论、代码、测试、部署和验收的各个环节，才能真正提高整个交付流程的效率。这才是我们真正追求的目标。

## 未来的应用构建工具会成为可控的AI工程平台

展望未来，AI 应用构建工具的发展路径，在我看来，大概会沿着几个引人入胜的方向展开。

**无缝协作：AI Agent融入工作流的每一个角落**

想象一下，AI agent 不再是孤立的工具，而是我们团队里的一员。它们将更自然地穿梭于 Issue、PR、设计稿、文档、监控系统和项目管理工具之间。这意味着开发者将不再需要频繁地在十几个窗口间切换，只为搬运那些零散的上下文。相反，这些 agent 可以在权限允许的范围内，主动且智能地读取所需材料，让我们的协作变得前所未有的流畅。

**全栈 Agent：从代码到产品，AI团队的雏形**

今天的许多 agent，虽然在代码生成方面表现出色，但对于产品、设计、数据、运维和安全这些领域的理解，仍然显得有些力不从心。未来的工具，我相信会更像一个由多角色组成的团队。比如，产品 agent 负责厘清需求，架构 agent 勾勒设计蓝图，开发 agent 负责将设想变为现实，测试 agent 严谨地验证成果，安全 agent 警惕地检查潜在风险，而发布 agent 则会细致地准备上线所需的一切。这不再是单一工具的智能，而是协作智能的涌现。

**低延迟与低成本：智能调度，让AI工程更持久**

要让 AI 工程长期可用，我们不能每次都依赖那些最昂贵、最耗时的大模型。未来的平台，必然会更加重视多模型的智能调度。简单任务，就交给小巧高效的模型；复杂的设计，才动用那些能力强大的模型；代码审查，则由专门的模型来把关；至于检索和索引，那些经济实惠的模型就能胜任。成本，将会成为我们工程架构中一个不可或缺的考量因素。

**可复用技能模块：团队经验的沉淀与复用**

我预感到，“技能”（Skills）会变得像团队内部的工具库一样重要。一个成熟的团队，会逐渐积累起自己独有的“测试技能”、“迁移技能”、“文档技能”、“组件规范技能”、“安全审查技能”等等。这些技能，不仅仅是代码片段，更是团队经验的结晶。它们能有效减少重复的提示工程，让 AI 的产出更贴近组织的内部标准，也更具团队特色。

**工程化可控：让AI的每一步都有迹可循**

AI 生成代码固然不难，但真正的挑战在于，如何让每一次变更都清晰可辨、可以审查、能够追溯，并且易于维护。未来的工具，必须提供更详尽的日志、更严格的权限控制、更可靠的回滚机制，以及更易于理解的变更记录。我们需要确保，AI 的每一步操作，都在我们的掌控之中。

**浏览器和真实环境验收：不只是编译通过，更是端到端可用**

未来的 AI 开发工具，绝不会满足于仅仅“代码编译通过”这样的表层成功。它会更强调端到端的真实验证：页面能否顺利打开？业务流程能否顺畅跑通？表单能否成功提交？错误提示是否准确呈现？接口返回是否符合预期？甚至视觉上的细微变化，是否都能被接受？浏览器将不再只是用户界面，它会成为 agent 的第二个工作台；而测试环境，则会化身为 agent 的专属实验室。

**协作系统回写：任务状态的自动同步与证据留存**

当 AI 完成任务后，它还需要智能地将结果回写到 Linear、Slack、GitHub、发布系统和文档系统等协作平台。状态的自动同步，将成为工程平台的核心功能之一。一个任务从“待处理”到“开发中”，再到“待审查”、“已验证”，直至“已发布”，每一个环节都应该有确凿的证据自动生成，而不是再依赖人工手动更新一句“已完成”。

最终，我猜想，AI Engineering Platform 将会成为我们新的开发基石。它固然包含代码编辑器，但远不止于此；它拥有 agent，但其能力远超单一 agent；它依赖模型，但又超越了模型的范畴。在我看来，它更像是一个宏大的工程系统，巧妙地将人、模型、工具、流程、知识和最终交付物，紧密地连接在一起。这是一个充满潜力的未来。


## 从Coding到AI Engineering，开发者的价值正在重新聚焦

说到底，从传统Coding到AI Engineering，我们经历的是一场重心转移。

曾几何时，衡量一个开发者的价值，往往看他写代码有多快，能记住多少API，对各种疑难杂症调优有多熟练。这些能力，今天依旧重要，但我们开始看到更高层次的价值正在浮现：你是否能准确地捕捉问题本质，是否能巧妙地拆解出可执行的任务，能否洞察AI方案潜在的风险，能否守住系统边界，甚至，能否将一堆看似独立的智能工具，编织成稳定的生产力。

> AI会让很多代码变得触手可及，但它也让那些草率的决策变得更加昂贵。毕竟，当产出速度加快，错误蔓延的速度也会同步提升。未来的优秀工程师，或许不再仅仅是那些代码写得漂亮的人，更是那些能设计工程环境、驾驭复杂性、定义质量标准，并巧妙调度AI劳动力的人。

这正是AI时代应用构建最引人入胜的地方。它既脚踏实地，关注着`worktree`、测试、部署、权限、成本、日志这些具体而微的细节；又带着一丝浪漫主义色彩，因为软件开发重新找回了那种探索的乐趣。你可以同时让几个agent尝试不同的实现路径，可以将一个模糊的想法迅速转化为可运行的原型，甚至在一天之内完成过去需要数周才能完成的早期探索。

然而，浪漫并不能取代严谨的工程。AI时代，真正可靠的应用构建，依然离不开清晰的架构、细致的验证、流畅的协作，以及人类不可或缺的判断力。AI拓宽了我们创造力的边界，但同时也放大了工程责任的重量。

所以，“从Coding到AI Engineering”绝非一句空泛的口号。它代表着工具、流程、角色乃至于思维方式的全方位迁徙。我们正在告别手工雕琢代码的时代，走向编排智能系统；从单打独斗的写作，转向人与agent的协同创作；从仅仅关注代码片段，到深入思考整个应用如何被持续、可靠、可控地生成出来。

> 未来的软件，其火种依旧会源自人类的意图。只不过，那团火焰不再仅仅照亮编辑器里闪烁的光标，它将照亮一整套正在协作的机器、流动的流程，以及无限的想象力。

想象一下，一个现代开发者坐在屏幕前，他面对的不再是一个孤零零的编辑器，而更像是一个小型机器工坊。左侧是版本历史的流转，右侧是文件结构的脉络，而屏幕中央，多个Agent正各司其职。或许有人在处理Linear上的任务，有人在翻阅Slack的通知，有人正忙着修改代码，有人在后台跑着测试，有人在部署预览环境，还有人在浏览器中进行最终验收。开发者在这里，不再仅仅是敲击键盘的执行者，他更像是一位指挥家，调度着这支智能工具的交响乐团，将一个最初模糊的需求，一点点塑造成真实可用的系统。

AI并没有让工程消失。相反，它将工程推向了一个更高的组织层面。因此，真正重要的能力，也从“我能写出多少代码”，扩展到了“我能否带领一支由模型、工具和流程组成的精干工程队，稳定地交付一个真正有价值的产品”。
