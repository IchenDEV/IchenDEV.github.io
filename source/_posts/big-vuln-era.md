# 我们已经站在「大漏洞时代」的入口

![](cover.jpg)

过去几年，安全行业常说「漏洞越来越多」。这句话听起来像老生常谈，可今天的变化更深。在写这篇文章前，我把近两年里让我印象深刻的安全事件整理了一遍，发现它们密集得有些让人不安：

| 时间 | 事件 | 类型 | CVE / 编号 | 严重程度 |
|------|------|------|------------|----------|
| 2022 年 | Dirty Pipe | Linux 内核本地提权 | CVE-2022-0847 | 高危，已被利用 |
| 2024 年 3 月 | XZ Utils 后门 | 供应链投毒 | CVE-2024-3094 | 严重，差点进主流发行版 |
| 2025 年 9 月 | npm 生态大规模供应链攻击 | 供应链投毒 | — | 高危，波及大量开发者 |
| 2025 年 | Nx 构建系统投毒 | 供应链投毒 | CVE-2025-10894 | 高危，窃取凭据 |
| 2026 年 3 月 24 日 | LiteLLM PyPI 投毒 | 供应链投毒 | — | 高危，针对 AI 工具链 |
| 2026 年 3 月 31 日 | Axios npm 投毒 | 供应链投毒 | — | 高危，RAT 植入 |
| 2026 年 5 月 1 日 | Copy Fail（Linux crypto 提权） | Linux 内核本地提权 | CVE-2026-31431 | 严重，已进入 CISA KEV |
| 2026 年 5 月 7 日 | Dirty Frag（Linux 网络子系统提权） | Linux 内核本地提权 | — | 高危，影响主流发行版 |
| 2026 年 5 月 11 日 | TanStack npm 供应链攻击 | 供应链投毒 | — | 高危，84 个恶意版本 |
| 2026 年 5 月 13 日 | NGINX rewrite 堆溢出 | 远程内存破坏 | CVE-2026-42945 | 严重（CVSS 9.2），可远程触发 |

这张表放在一起看，会有一种奇怪的感觉：它们来自不同的方向，内核、包管理器、AI 工具链、Web 服务器，但节奏越来越密，影响越来越广。这不是偶然的波动，而是一个趋势正在成形。

真正值得警惕的地方在于，AI 正在降低发现基础软件漏洞的成本，而基础软件又支撑着云、容器、开发工具、CI/CD、企业内网和终端设备。一旦这里出现问题，影响不会停留在某个应用或某台机器上，它会沿着软件依赖、镜像、服务器、开发者电脑和自动化流水线一路扩散。

我把这个阶段称为「大漏洞时代」。

它的特征有三个。第一，漏洞不再只集中在业务系统，越来越多地出现在 Linux 内核、密码学接口、包管理器、开源依赖和构建链路里。第二，漏洞利用变得更稳定，很多提权漏洞不再依赖复杂的时机竞争，普通本地用户也可能快速拿到 root。第三，供应链攻击从「偶发事故」变成高频风险，只要攻击者控制一个维护者账号、一个 npm 包、一个构建脚本，就可能影响成千上万的项目。

## AI 让基础软件漏洞更容易被发现

![](chapter1.jpg)

AI 对安全攻防最大的影响，未必是自动写钓鱼邮件，也未必是生成恶意代码。更关键的是，它开始参与「读代码」这件事。

基础软件的代码过去很难审。Linux 内核、OpenSSL、glibc、npm 生态、容器运行时、系统服务，这些项目的代码量大、历史长、边界复杂。一个危险逻辑可能藏在多年以前的提交里，也可能只在几个系统调用、共享内存、页缓存或权限检查组合在一起时才显现出来。人类研究员当然可以发现这些问题，但成本很高，速度很慢。

现在，AI 可以帮助研究员缩小搜索范围，梳理调用路径，提示看似正常但权限语义不一致的地方。Copy Fail 就是一个很有代表性的信号。Theori / Xint 的资料显示，他们让 Xint Code 分析 Linux crypto 子系统中从用户态系统调用可达的代码路径，约一小时后，Copy Fail 成为最高严重度输出之一。这个漏洞后来被编号为 CVE-2026-31431，并进入 CISA 已知被利用漏洞目录。

这件事的意义不在于「AI 一定比人更聪明」。它说明漏洞研究的成本结构变了。过去需要少数专家长时间阅读的代码，现在可以被工具大规模扫描。攻击者也能使用类似思路去找问题。防守方如果仍然只靠人工巡检、偶尔升级、出了事再补丁，就会越来越吃力。

## Linux 提权漏洞说明基础设施的安全边界正在变薄

![](chapter2.jpg)

Linux 是现代互联网的底座。云服务器、Kubernetes 节点、容器宿主机、边缘设备、NAS、手机和大量开发环境都依赖它。正因为它足够基础，一个 Linux 内核本地提权漏洞的真实影响，往往远超「本地漏洞」四个字。

Dirty Pipe 是这个谱系里的经典案例。它的编号是 CVE-2022-0847，属于 Linux 内核本地提权漏洞。NVD 页面显示，它已经被纳入 CISA 已知被利用漏洞目录；许多安全团队将它视为 Linux 页缓存相关漏洞的一个重要节点。

Copy Fail 则把问题推得更近一步。NVD 将 CVE-2026-31431 描述为 Linux Kernel「Incorrect Resource Transfer Between Spheres」漏洞，并显示 CISA 在 2026 年 5 月 1 日将它加入已知被利用漏洞目录，要求按厂商指引处理或停用受影响产品。Red Hat 也提示该漏洞已有公开利用代码，应高优先级处理。

更近的例子是 Dirty Frag。2026 年 5 月 7 日到 5 月 8 日，多家发行版和安全机构开始发布 Dirty Frag 相关公告。Red Hat 安全公告称，Dirty Frag 涉及 Linux 内核网络子系统中的两个问题，本地账号用户可能利用这些问题获得 root 权限。Openwall 的披露邮件也称，Dirty Frag 是一个影响主要 Linux 发行版的通用本地提权问题。

这几个案例放在一起看，就会看到一个危险趋势：攻击者一旦拿到普通权限，下一步走向 root 的距离正在缩短。容器、最小权限、普通用户账号，这些设计仍然有价值，但它们不能再被当成牢固的最后屏障。基础软件漏洞会让这些边界变薄，甚至让看似隔离的环境互相影响。

## NGINX 也没能幸免：CVE-2026-42945 与基础设施的暴露面

内核之外，另一个方向的风险同样值得关注。NGINX 是互联网上部署最广泛的 Web 服务器和反向代理之一，几乎出现在每一条流量路径上。2026 年 5 月 13 日，F5 发布安全公告，披露了 CVE-2026-42945，漏洞由安全研究员 Zhenpeng (Leo) Lin 发现并负责任地披露。

这个漏洞存在于 NGINX 的 `ngx_http_rewrite_module` 模块中。当 `rewrite` 指令后跟 `rewrite`、`if` 或 `set` 指令，且使用了未命名的 PCRE 捕获组（如 `$1`、`$2`），同时替换字符串中包含问号（`?`）时，漏洞就会触发。未经认证的攻击者通过发送精心构造的 HTTP 请求，可以在 NGINX worker 进程中触发堆缓冲区溢出，导致进程重启。更危险的是，对于禁用了地址空间布局随机化（ASLR）的系统，漏洞还可能进一步实现代码执行。

F5 的安全公告将该漏洞评级为 CVSS v3.1 高危（8.1）、CVSS v4.0 严重（9.2）。受影响范围相当广泛：NGINX Plus R32 至 R36、NGINX Open Source 1.0.0 至 1.30.0，以及 NGINX Ingress Controller、NGINX Gateway Fabric、NGINX App Protect WAF 等一系列周边产品均受波及。修复版本为 NGINX Plus R36 P4 / R32 P6，以及 NGINX Open Source 1.31.0 / 1.30.1。

这个漏洞的意义不只在于它本身的严重程度。NGINX 的特殊位置——几乎所有云原生架构的流量入口——意味着这里一旦出现可远程触发的内存破坏漏洞，攻击面就会直接暴露在互联网上，不需要任何前置的本地权限。这与 Linux 内核提权漏洞需要先拿到本地立足点的逻辑完全不同，它是从外部直接击打基础设施的核心节点。

## 「本地漏洞」在云和容器时代不再只是本地问题

很多人看到「本地提权」会下意识松一口气，觉得攻击者必须先登录机器，风险自然较低。这个想法在今天已经过时了。

在云和容器环境里，「拿到一个低权限入口」并不罕见。一次 Web 漏洞、一个暴露的开发服务、一个泄露的 CI token、一个被污染的镜像，都可能让攻击者进入某个容器或普通用户上下文。此时，本地提权漏洞就会成为第二段攻击链。它把「低权限立足点」变成「宿主机 root」「横向移动入口」或「窃取密钥的机会」。

这也是 Dirty Pipe、Copy Fail、Dirty Frag 这类漏洞让人紧张的原因。它们触及的是 Linux 内核中极底层的文件、页缓存、网络或密码学处理路径。只要攻击者能在受影响系统上执行代码，提权就可能成为现实。CISA 将 Copy Fail 加入 KEV，也说明这类漏洞不只存在于实验室里，而是已经进入真实攻击视野。

## 软件供应链已经成为攻击者更喜欢走的路

![](chapter3.jpg)

和内核漏洞并行发展的，是软件供应链攻击的高频化。

供应链攻击的可怕之处在于，它利用的是开发者的信任。开发者相信 npm、PyPI、GitHub、Docker 镜像、CI/CD 插件、开源维护者账号和自动升级流程。攻击者只要污染其中一个环节，就可能让恶意代码以「正常依赖」的身份进入项目。

Axios 事件就是一个典型例子。Axios 官方 GitHub 事后说明显示，2026 年 3 月 31 日，两个恶意版本 axios 1.14.1 和 0.30.4 通过被攻陷的维护者账号发布到 npm；这两个版本引入了 plain-crypto-js 4.2.1，并在 macOS、Windows 和 Linux 上安装远程访问木马。官方说明还提到，恶意版本大约在线三小时后被移除。

类似事件一直在增多。2024 年的 XZ Utils 后门是另一个极端案例。CISA 在 2024 年 3 月 29 日发布警告称，XZ Utils 的特定版本中发现恶意代码，可能允许未授权访问受影响系统；NVD 对 CVE-2024-3094 的描述也指出，恶意代码被插入到上游 tarball 中，并通过复杂混淆影响 liblzma 构建过程。

2025 年，npm 生态也出现了大规模供应链事件。CISA 在 2025 年 9 月 23 日发布关于 npm 生态供应链攻击的提醒，称攻击者利用自动化过程，通过受害开发者身份认证到 npm 注册表，并向其他包注入代码。Nx 事件也被记录为 CVE-2025-10894，NVD 描述称恶意代码被插入 Nx 构建系统包及相关插件，发布到 npm 后会扫描文件系统、读取凭据并上传。

2026 年 3 月 24 日，LiteLLM 又给 AI 基础设施敲了一次警钟。[LiteLLM 官方安全更新](https://docs.litellm.ai/blog/security-update-march-2026)称，PyPI 上的 `litellm==1.82.7` 和 `litellm==1.82.8` 是受影响版本，两个包从 10:39 UTC 开始可用，约 40 分钟后被 PyPI 隔离；官方认为事件可能来自 CI/CD 安全扫描流程里使用的 Trivy 依赖。1.82.7 在 LiteLLM AI Gateway 的 `proxy_server.py` 里带恶意 payload，1.82.8 又加入 `litellm_init.pth`，让 Python 环境启动时就可能执行恶意逻辑。

这个案例尤其值得注意，因为 LiteLLM 这类网关通常靠近 LLM API key、云凭据、Kubernetes token 和数据库密码。官方说明称，恶意代码会读取环境变量、SSH key、云厂商凭据、Kubernetes token 和数据库密码，并把数据加密后 POST 到非官方域名 `models.litellm.cloud`。它提醒开发者：AI 工具链里的辅助组件，只要进入构建、部署或运行环境，就可能接触到比业务代码更敏感的权限。

2026 年 5 月 11 日的 TanStack 事件把这条链路讲得更清楚。[TanStack 官方复盘](https://tanstack.com/blog/npm-supply-chain-compromise-postmortem)称，攻击者组合了 `pull_request_target` 的高风险用法、GitHub Actions cache poisoning 和 runner 运行时里的 OIDC token 提取，最终向 npm 发布了 42 个 `@tanstack/*` 包的 84 个恶意版本。TanStack 特别说明，npm token 没有被盗，npm 发布 workflow 本身也没有被攻破；真正的问题在于 fork PR、构建缓存、发布身份和 runner 运行时之间的信任边界被连续跨过了。

更麻烦的是，恶意包会在 `npm install`、`pnpm install` 或 `yarn install` 时通过 lifecycle script 运行，读取云凭据、Kubernetes service-account token、Vault token、npm/GitHub token 和 SSH 私钥，并通过 Session/Oxen 网络外传。也就是说，风险不只在 registry 上的几个版本号里，还在安装环境里的凭据、CI 权限和发布权限里。

这些事件说明，软件供应链已经从「代码质量问题」升级为「身份、发布、构建、依赖、运行环境」一起承受的风险。开发者安装一个包，CI 拉取一个版本，镜像构建执行一个 postinstall 脚本，都可能成为攻击入口。

## 大漏洞时代的核心风险是「发现更快，扩散也更快」

今天的漏洞风险变大，并不只是因为漏洞数量增加。真正的问题是速度。

AI 让漏洞发现更快。开源生态让漏洞影响更广。云和容器让一个漏洞可以穿过更多边界。自动化构建和自动更新让恶意版本传播更快。开发者本地机器、CI runner、生产服务器和云密钥又常常连在一起，任何一个环节出问题，都可能影响后面整条链。

这就是「大漏洞时代」的核心：漏洞从单点事件变成链式事件。

一个 Linux 内核漏洞，可以把普通权限变成 root。一个 npm 包被污染，可以把木马送进开发者电脑和 CI/CD。一个维护者账号被攻陷，可以让可信项目短时间内发布恶意版本。一个旧依赖未锁定，可以在下一次构建时拉入危险代码。

过去，我们常把安全看成「上线前检查」。现在，这种思路已经不够。安全必须进入持续运行状态，覆盖依赖选择、版本锁定、构建隔离、密钥管理、补丁节奏、运行时监控和事故响应。

## 我们应该怎么办

大漏洞时代里，最危险的心态是相信某个系统足够成熟，所以不会出大问题。Linux 足够成熟，还是会有 Dirty Pipe、Copy Fail、Dirty Frag。Axios 足够流行，还是会出现被污染版本。XZ Utils 足够底层，还是差点把后门带进关键系统。

更现实的前提是：基础软件会出漏洞，维护者账号会被攻陷，依赖会被污染，CI 会执行恶意脚本，密钥会暴露，攻击者会把多个小入口串起来。安全工作的目标要更具体：出事时尽量把影响限制在小范围内，并且能快速判断哪些系统需要处理。

第一，管住依赖。重要项目必须有 lockfile，生产构建不要用无版本约束的 `npm install`、`pip install -U` 或 `pip install 包名`。升级热门依赖时，先看 release、tag、官方公告和仓库状态；遇到短时间内突然冒出的版本，尤其是没有 GitHub tag 或 changelog 的版本，先停一下。对 npm、PyPI 这类生态，能关 lifecycle script 的地方就关，必须打开时让它在隔离环境里跑。

第二，降低 CI 和开发机的含权量。CI token 用最小权限，PR 工作流不要默认拿发布权限，fork PR 和发布任务要分开。云密钥、npm token、PyPI token、SSH 私钥不要长期放在开发机和 runner 上；能用 OIDC 短期凭据就用短期凭据。构建环境最好一次性使用，跑完销毁，避免一个被污染的包带走长期凭据。

第三，把事故处理前置。依赖升级后出现异常网络请求、内存暴涨、奇怪的 `.pth` 文件、postinstall 脚本、未知 systemd service、陌生 Kubernetes pod，就按凭据可能泄露处理。先隔离环境，再查版本和文件，再轮换相关 token。不要只把包降级就结束，因为恶意代码可能已经读过环境变量和本地文件。

第四，减少盲目信任。开源包可以信任，但信任要配边界：固定版本、校验来源、分离权限、记录构建产物、保留依赖变更日志。对靠近 AI API key、云账号、CI 发布权限的工具，要用更高标准审视。它们看起来只是辅助库，实际常常站在权限最集中的位置。

我们做不到完全避开漏洞，但可以让漏洞少拿到东西、少进入系统、少停留时间。一旦把这件事当成日常工程纪律，而非事故后的临时补救，大漏洞时代的风险就不会每次都直达核心资产。

## 大漏洞时代已经开始

![](ending.jpg)

AI 发展会让漏洞发现能力进一步放大。基础软件越复杂，隐藏多年的逻辑错误越可能被重新翻出来。与此同时，供应链攻击会继续瞄准维护者账号、包发布流程、构建脚本和开发者机器。攻击者越来越少正面硬打生产系统，越来越多从开发和依赖入口进入。

Dirty Pipe、Copy Fail、Dirty Frag 提醒我们，底层系统的一个逻辑细节，可能击穿 root 边界。CVE-2026-42945 提醒我们，流量入口同样不是安全地带——NGINX 的一个 rewrite 模块缺陷，可以让未经认证的攻击者从互联网直接触达服务进程。Axios、XZ Utils、Nx、LiteLLM、TanStack 和 npm 供应链事件提醒我们，可信依赖也可能在短时间内变成攻击载体。

我们即将站在大漏洞时代。更准确地说，我们已经站在门口了。
